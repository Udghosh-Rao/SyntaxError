This document is the single source of truth for building the Sports Event Management

1. Technical Specification – Every model, relationship, API endpoint, page, com-

be built with Flask and Flask-RESTful. The frontend must be built with Vue 3.
Every data model, API endpoint, and UI page listed in this document must be
implemented completely and correctly.
1.2
Problem Statement
The sports event industry lacks a unified digital platform that efficiently connects event
organizers with participants while providing intelligent recommendations, real-time an-
alytics, and seamless payment processing. Organizers struggle to monitor event perfor-
mance, administrators lack platform-wide visibility, and users face difficulty discovering
relevant local events.
This platform solves all three problems by delivering a single web application with three
distinct role-based interfaces, intelligent recommendation logic, BI dashboards, Razorpay
payments, Algolia search, and a LangChain-powered AI chatbot.

2.
Technology Stack
2.1
Backend Technologies
The backend is a RESTful API server built on the Python ecosystem.
Flask is the core web framework. Flask-RESTful is used to define all API resources as
class-based views with clean HTTP method separation. SQLAlchemy is the ORM layer;
it maps Python classes to database tables and manages all queries. Flask-Migrate (built
on Alembic) handles all database schema migrations. Flask-JWT-Extended manages
JSON Web Token authentication — every protected endpoint verifies a Bearer token in the
Authorization header. Flask-CORS enables cross-origin requests from the Vue frontend.
The development database is SQLite and the production database is PostgreSQL.
2.2
Frontend Technologies
The frontend is a Single Page Application (SPA) built with Vue 3 CLI. Vue Router
handles client-side routing between pages without full page reloads. Pinia is the global
state management library — it stores the authenticated user’s token, role, and ID across all
components. Axios is the HTTP client used to call all backend API endpoints. Chart.js
with the vue-chartjs wrapper renders all analytics charts (line graphs and bar charts).
The Razorpay JavaScript SDK is loaded via a CDN script tag to open the payment
checkout modal.
2.3
GenAI Technologies
The AI chatbot is built using LangChain as the orchestration framework. LangChain
connects to OpenAI GPT-4o-mini as the primary LLM. As fallback options, Google
Gemini and HuggingFace inference models may also be used. A FAISS vector store
holds the FAQ knowledge base as embeddings. The chatbot uses a Retrieval-Augmented
Generation (RAG) pipeline: incoming queries are matched against the vector store and
the top matching FAQ chunks are injected into the LLM prompt as context.
2.4
Third-Party Services
Algolia provides instant, typo-tolerant full-text search over all events. Event records are
synced to Algolia any time an event is created or updated. The Vue frontend uses the
Algolia JavaScript client directly for search queries.

4.
User Roles & Access Control
Agent Rule #4: Every backend API endpoint must enforce role-based access con-
trol. A custom decorator called role_required(role) must be implemented and
applied to every protected route. If a user’s JWT role does not match the required
role, return HTTP 403 Forbidden.
4.1
Role Definitions
The platform has exactly three user roles, all stored in the role field of the User model:
4.1.1
Role: user
A standard end-user who discovers and registers for events. After login, they are redirected
to the Home / Event Discovery page. They can browse recommended events, search
events via Algolia, view event details, register for events, pay via Razorpay, and use the
AI chatbot.
4.1.2
Role: organizer
An event organizer who creates and manages sports events. After login, they are redirected
to the Organizer Dashboard. They can create, edit, and delete their own events. They
can also view their event performance metrics, registration trends, ticket sales summaries,
and category insights. They cannot see other organizers’ data and cannot access admin
features.
4.1.3
Role: admin
A platform administrator. After login, they are redirected to the Admin Dashboard.
They have read access to all platform data including total users, all events, all registra-
tions, city distributions, monthly trends, and organizer performance comparisons. They
also receive and resolve chatbot escalation tickets. They cannot be registered through the
public registration form — admin accounts must be seeded directly into the database.

9.
Payment Flow Specification
Agent Rule #9: Implement the Razorpay payment flow exactly as described in
the five steps below. Do not allow a user to gain a confirmed registration without a
verified payment signature.
9.1
End-to-End Payment Flow
The complete flow from user intent to confirmed registration is:
1. The user views an Event Detail page and clicks Register & Pay.
2. The Vue frontend sends a POST request to /api/payments/create-order with the
event_id. The backend creates a Razorpay order via the Razorpay Python SDK,
creates a Registration record (status: pending), and a Payment record (status:
created).
The backend returns order_id, amount, currency, and key_id to the
frontend.
3. The frontend opens the Razorpay checkout modal using the returned values. The
user completes payment via their preferred method (card, UPI, wallet, or net bank-
ing).
4. Razorpay calls the frontend handler with three values: razorpay_order_id, razor-
pay_payment_id, and razorpay_signature.
5. The frontend immediately sends a POST request to /api/payments/verify with
the three Razorpay values. The backend recomputes the HMAC-SHA256 signature
using the secret key and verifies it matches. On success, the Payment status is
updated to paid and the Registration status is updated to confirmed. The orga-
nizer_payout is stored as the amount minus the platform fee (15–20%). The user
sees a success message.
9.2
Platform Fee Calculation
The platform fee percentage is stored in the application config as PLATFORM_FEE_PERCENT
(default 0.15, representing 15%). The organizer_payout stored in the Payment record
equals:
organizer_payout = amount × (1 −PLATFORM_FEE_PERCENT)

10.
AI Chatbot Specification
Agent Rule #10: Build the chatbot as a LangChain Retrieval-Augmented Gen-
eration (RAG) pipeline. The FAISS vector store must be initialised at application
startup with the FAQ documents listed below. Do not regenerate embeddings on
every request — cache the vector store in memory.
10.1
RAG Pipeline Architecture
When a user sends a message to the chatbot endpoint, the following steps occur in se-
quence inside the backend service:
1. The user’s query is received as plain text.
2. The LangChain retriever searches the FAISS vector store for the top 3 most seman-
tically similar FAQ chunks.
3. The retrieved chunks are injected as context into a PromptTemplate alongside the
user’s question.
4. The assembled prompt is sent to the LLM (OpenAI GPT-4o-mini).
5. The LLM generates an answer grounded in the FAQ context.
6. The answer is returned to the user via the API response.
7. If no relevant FAQ context is found (retrieval score is below threshold), the chatbot
responds with a standard escalation message and saves an EscalationTicket record
to the database.
10.2

startup. More can be added over time via a database-backed FAQ management system:
• How do I register for an event? Navigate to the event detail page and click Register

Deployment Checklist
Before going live, complete the following steps in order:
1. Set all environment variables in the hosting platform’s secret manager.
2. Run flask db upgrade against the production PostgreSQL database to apply all
migrations.
3. Seed the Algolia index by triggering the sync service for all existing events.
4. Set the Razorpay account to Live Mode and update the key variables.
5. Deploy the Vue frontend with the production API base URL configured.
6. Run a smoke test on all critical flows: register, login, browse events, register for
event, pay, view organizer dashboard, view admin dashboard, use chatbot.
